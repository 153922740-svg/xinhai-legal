## 一、分支管理

### 1.1 分支类型

| 分支 | 用途 | 保护级别 | 命名规范 |
|------|------|----------|----------|
| **main** | 生产环境代码 | 严格保护 | main |
| **staging** | 预发布环境代码 | 保护 | staging |
| **develop** | 开发分支 | 保护 | develop |
| **feature** | 功能开发 | 自行管理 | feature/功能名 - 开发者 |
| **hotfix** | 紧急修复 | 保护 | hotfix/问题描述 - 日期 |

### 1.2 分支生命周期

```
main (生产)
  ↑
staging (预发布)
  ↑
develop (开发)
  ↑
feature/* (功能开发) → 删除
```

### 1.3 分支创建

**feature 分支**:
```bash
# 从 develop 创建功能分支
git checkout develop
git pull
git checkout -b feature/user-auth-coo
```

**hotfix 分支**:
```bash
# 从 main 创建紧急修复分支
git checkout main
git pull
git checkout -b hotfix/login-fix-20260518
```

## 二、提交规范

### 2.1 提交信息格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 2.2 type 类型

| type | 用途 | 示例 |
|------|------|------|
| **feat** | 新功能 | feat(auth): 添加微信登录 |
| **fix** | Bug 修复 | fix(login): 修复验证码问题 |
| **docs** | 文档更新 | docs(api): 更新接口文档 |
| **style** | 格式调整 | style: 代码格式化 |
| **refactor** | 代码重构 | refactor: 重构认证模块 |
| **test** | 测试相关 | test: 添加登录测试 |
| **chore** | 构建/工具 | chore: 更新依赖 |

### 2.3 scope 范围

| scope | 说明 |
|-------|------|
| auth | 认证模块 |
| member | 会员模块 |
| payment | 支付模块 |
| document | 文书模块 |
| chat | 对话模块 |
| partner | 合伙人模块 |
| integral | 积分模块 |

### 2.4 subject 要求

- 简短描述（不超过 50 字）
- 使用动词开头
- 不用句号结尾

**示例**:
```
feat(auth): 添加微信登录功能
fix(login): 修复验证码发送失败
docs(api): 更新认证接口文档
```

### 2.5 完整示例

```
feat(auth): 添加微信登录功能

- 实现微信授权登录
- 添加 openid 缓存
- 完善错误处理

Closes #123
```

## 三、PR 流程

### 3.1 PR 提交流程

```
开发完成 → 自测通过 → 提交 PR → CI 检查 → 代码审查 → 修复问题 → 审查通过 → 合并
```

### 3.2 PR 要求

**提交前检查**:
- [ ] 代码已自测通过
- [ ] 测试用例已编写
- [ ] 文档已更新
- [ ] 提交信息规范
- [ ] 无合并冲突

**PR 描述模板**:
```markdown
## 变更说明
（描述本次 PR 的变更内容）

## 测试说明
（描述如何测试本次变更）

## 相关 Issue
（关联的 Issue 编号）

## 检查清单
- [ ] 代码已自测
- [ ] 测试用例已添加
- [ ] 文档已更新
- [ ] 无合并冲突
```

### 3.3 PR 审查

**审查人**:
- 技术审查：铸基
- 代码审查：明鉴
- 最终审批：COO

**审查要点**:
- 代码规范
- 功能完整性
- 错误处理
- 性能影响
- 安全风险

**审查时限**:
- P0 PR: 2 小时内
- P1 PR: 24 小时内
- P2 PR: 72 小时内

### 3.4 PR 合并

**合并条件**:
- [ ] CI 检查通过
- [ ] 审查通过
- [ ] 无合并冲突
- [ ] 测试通过

**合并方式**:
- **Squash and Merge**: 多个提交压缩为一个
- **Rebase and Merge**: 变基后合并
- **Create a Merge Commit**: 创建合并提交

## 四、冲突解决

### 4.1 冲突预防

- 及时 pull 最新代码
- 小步提交，频繁同步
- 避免同时修改同一文件
- 沟通好修改范围

### 4.2 冲突解决流程

```
发现冲突 → pull 最新代码 → 本地解决冲突 → 测试通过 → 提交解决
```

### 4.3 冲突解决工具

- **VS Code**: 内置冲突解决工具
- **Git**: 命令行解决
- **GitKraken**: 图形化工具

## 五、回滚流程

### 5.1 回滚场景

- 上线后发现严重 Bug
- 功能不符合需求
- 性能问题严重

### 5.2 回滚流程

```
发现需要回滚 → COO 审批 → 创建 hotfix 分支 → 回滚代码 → 测试验证 → 合并到 main → 部署
```

### 5.3 回滚命令

```bash
# 回滚上一次提交
git revert HEAD

# 回滚指定提交
git revert <commit-id>

# 强制回滚到指定版本
git reset --hard <commit-id>
```

## 六、代码审查

### 6.1 审查清单

**代码规范**:
- [ ] 符合 PEP8 规范

- [ ] 代码简洁

**功能完整性**:

- [ ] 无性能问题

- [ ] 缓存使用合理

**测试**:
- [ ] 单元测试覆盖
- [ ] 测试用例完整

```markdown

⬜ 通过
⬜ 有条件通过（需修改上述问题）
⬜ 不通过（需要重新开发）
```

## 七、Git 纪律

### 7.1 提交纪律

- **真实性**: 提交内容真实，不弄虚作假
- **及时性**: 完成功能及时提交
- **规范性**: 提交信息符合规范
- **原子性**: 一次提交一个功能点

### 7.2 分支纪律

- **不直接在 main 开发**: 禁止直接在 main 分支开发
- **及时删除 feature 分支**: 合并后及时删除
- **保护分支不强制推送**: 禁止强制推送到保护分支

### 7.3 违规处理

| 违规行为 | 处理措施 |
|----------|----------|
| 提交信息不规范 | 口头警告 + 修改 |
| 直接在 main 开发 | 书面警告 |
| 强制推送保护分支 | 严重警告 |
| 代码审查不认真 | 口头警告 + 重新审查 |
