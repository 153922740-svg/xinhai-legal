## 核心流程（必须严格执行）

### 两阶段评审

**阶段 1 开发前评审**:
```
填写接口设计文档 → COO 初审 → 技术评审 (铸基) → 前端评审 (匠心) → 测试评审 (铁壁) → COO 审批 → 开始开发

**阶段 2 开发后验收**:
```
开发完成 → 36 项自检 → 提交代码 → 评审会现场演示 → 实际测试 → 评审通过 → 代码合并

**关键改进**:
- 从"先开发后补票" → "先设计后开发"
- 从"形式评审" → "实测评审"（现场演示 + 实际测试）
- 从"凭感觉检查" → "清单式检查"（36 项自检）

## 36 项自检清单（必须全部通过）

### 一、代码实现（10 项）
- 是否添加 phaseX_v1_bp Blueprint
- Blueprint 前缀是否为 /api/v1/模块名
- 路由路径、方法是否正确
- v1 路由函数是否调用原函数
- 函数名是否正确（无拼写错误）
- 是否在 app.py 中导入和注册 v1 Blueprint
- 代码是否符合 PEP8 规范
- 错误处理是否完善

### 二、功能验证（8 项）
- 本地测试通过（curl 测试）
- 响应格式正确
- 响应内容符合预期
- 错误处理正常
- 域名测试通过
- Nginx 转发正确
- SSL 证书有效
- 性能达标（响应时间<2s）

### 三、文档同步（12 项）
- 更新接口文档 V2.0
- 包含请求参数说明、请求示例、响应示例、错误码说明
- 标注小程序调用路径和后端实际路径
- 更新接口注册表
- 填写接口设计文档
- 开发后验收已填写、开发者已签名

### 四、提交确认（6 项）
- Git 提交信息规范
- 提交文件完整、已推送
- 接口设计文档、测试报告准备就绪
- 已预约评审会时间

**自检要求**: 36 项全部通过才能提交评审，自检清单必须开发者签名

## 红线（违反=严重失误）

1. 未经评审先开发 - 书面警告 + 重新评审
2. 未经测试先上线 - 通报批评 + 绩效扣分
3. 擅自修改生产 - 严重警告 + 权限回收
4. 文档不同步 - 口头警告 + 立即整改
5. 自检清单未填写 - 口头警告 + 立即补填
6. 现场演示测试失败 - 重新开发 + 重新评审
7. Bug 隐瞒不报 - 通报批评 + 绩效扣分
8. 培训考核不通过 - 补考 + 口头警告
9. 泄露敏感信息 - 严重警告 + 辞退
10. 直接 main 分支开发 - 书面警告
