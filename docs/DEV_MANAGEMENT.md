## 一、环境管理

### 1.1 环境划分

| 环境 | 用途 | 服务器 | 访问方式 |
|------|------|--------|---------|
| **开发环境** | 日常开发、单元测试 | 本地/开发服务器 | 开发者访问 |
| **测试环境** | 集成测试、QA 验证 | 8.218.93.213 (测试实例) | 测试团队访问 |
| **预发布环境** | 上线前验证 | 8.218.93.213 (staging) | 核心团队访问 |
| **生产环境** | 线上服务 | 8.218.93.213 | 用户访问 |

### 1.2 环境隔离原则

- ❌ 禁止在生产环境直接修改代码
- ✅ 所有修改必须先在开发环境完成
- ✅ 测试通过后才能部署到生产

# 生产环境保持不变
/www/wwwroot/xinclaw-law/

/home/admin/xinclaw-code/miniprogram/

## 三、开发流程

### 3.1 标准开发流程

```
[需求提出] → [需求评审] → [技术设计] → [设计评审]
     ↓
[开发环境编码] → [单元测试] → [代码审查]
     ↓
[测试环境部署] → [集成测试] → [测试报告]
     ↓
[预发布验证] → [上线审批] → [生产部署]
     ↓
[线上监控] → [问题追踪] → [版本发布]
```

### 3.2 各阶段准入准出标准

| 阶段 | 准入标准 | 准出标准 |
|------|---------|---------|
| 开发 | 需求评审通过 | 单元测试通过 |
| 测试 | 代码审查通过 | 测试报告通过 |
| 部署 | 测试报告通过 | 线上验证通过 |

### 3.3 紧急修复流程

```
[问题发现] → [问题评估] → [COO 审批]
     ↓
[开发环境修复] → [快速测试] → [审批]
     ↓
[生产部署] → [问题验证] → [补充文档]
```

## 四、版本管理

### 4.1 Git 使用规范

```
分支管理:
- main: 生产分支（受保护）
- develop: 开发分支
- feature/*: 功能分支
- hotfix/*: 紧急修复分支
```

### 4.2 提交规范

```
feat: 新功能
fix: 修复 bug
docs: 文档更新
style: 代码格式
refactor: 重构
test: 测试
chore: 构建/工具
```
