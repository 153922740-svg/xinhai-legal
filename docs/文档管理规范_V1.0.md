## 二、文档命名规范

### 2.1 文件名格式

```
<文档类型>_<文档名称>_V<版本号>.md
```

**示例**:
- `接口设计文档_用户登录_V1.0.md`
- `测试报告_会员系统_V2.0.md`
- `会议纪要_评审会_20260518.md`

### 2.2 版本号规则

- **V1.0**: 初版
- **V1.1**: 小修订（文字修改、错别字）
- **V2.0**: 大修订（内容重大变更）
- **V2.1**: 大修订后的小修订

### 2.3 日期格式

- **文件名中的日期**: YYYYMMDD（如 20260518）
- **文档内日期**: YYYY-MM-DD（如 2026-05-18）

## 三、文档版本管理

### 3.1 Git 管理

**分支策略**:
- `main`: 生产文档（已发布）
- `develop`: 开发文档（进行中）
- `feature/xxx`: 功能文档（草稿）

**提交流程**:
```
编写文档 → 提交到 feature 分支 → 评审通过 → 合并到 develop → 发布后合并到 main
```

### 3.2 提交信息规范

```
docs(<文档类型>): <修改说明>

- 修改内容 1
- 修改内容 2
```

**示例**:
```
docs(接口文档): 更新用户认证接口文档

- 添加 v1 兼容路由说明
- 更新请求示例
- 添加错误码说明
```

### 3.3 版本历史

每个文档必须包含版本历史:

```markdown
## 版本历史

| 版本 | 日期 | 修改人 | 修改说明 |
|------|------|--------|----------|
| V1.0 | 2026-05-18 | COO | 初版 |
| V1.1 | 2026-05-19 | COO | 修订错别字 |
| V2.0 | 2026-05-20 | COO | 重大更新 |
```

## 七、文档管理纪律

### 7.1 编写要求

- **及时性**: 文档与代码同步编写
- **准确性**: 内容准确，无错误
- **完整性**: 内容完整，无遗漏
- **规范性**: 符合命名和格式规范

### 7.2 更新要求

- **同步性**: 代码变更后立即更新文档
- **版本控制**: 更新必须升级版本号
- **版本历史**: 记录每次修改内容

### 7.3 违规处理

| 违规行为 | 处理措施 |
|----------|----------|
| 文档与代码不同步 | 口头警告 + 立即更新 |
| 文档命名不规范 | 口头警告 + 立即修改 |
| 文档未版本管理 | 口头警告 + 补充版本 |
| 文档内容错误 | 书面警告 + 立即修正 |
