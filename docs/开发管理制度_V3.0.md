## 第一章 总则

- **规范开发流程** - 确保开发工作有序、高效

## 第二章 接口管理制度

### 2.1 接口设计文档制度

**原则**: 先设计，后开发。未经设计评审，不得开始开发。

**流程**: 填写接口设计文档 → 评审会通过 → 开始开发

**接口设计文档内容**:
1. 基本信息：接口名称、路径、方法、Phase、负责人
2. 请求参数：参数名、类型、必填、默认值、说明、示例
3. 响应格式：成功响应、错误响应、错误码
4. 后端设计：v1 兼容路由代码、处理函数
5. 测试设计：测试用例、测试命令
6. 评审记录：开发前评审、开发后验收

**模板**: `docs/接口设计文档模板.md`

**违规**: 未经设计评审直接开发 = 书面警告 + 重新评审

### 2.2 接口评审制度

**评审组织**: COO(组长)+铸基 (技术)+匠心 (前端)+铁壁 (测试)

**两阶段评审**:
- **阶段 1 开发前评审**: 接口设计文档评审 → COO 审批 → 开始开发
- **阶段 2 开发后验收**: 开发完成 → 36 项自检 → 现场演示 → 实际测试 → 评审通过 → 代码合并

**评审内容**: 路径 (/api/v1/*)、参数、响应、错误码、安全性、v1 兼容、文档、测试用例

**评审方式**:
- 开发前评审：文档评审
- 开发后验收：现场演示 + 实际测试（必须）

**时限**: 核心 24h/重要 48h/一般 72h

**红线**: 未经评审的接口一律不准上线（违反=严重失误）

### 2.3 接口开发自检制度

**原则**: 开发完成后，开发者必须先自检，全部通过才能提交评审。

**自检清单（36 项）**:

**一、代码实现（10 项）**:
- 是否添加 phaseX_v1_bp Blueprint
- Blueprint 前缀是否为 /api/v1/模块名
- 路由路径、方法是否正确
- v1 路由函数是否调用原函数
- 函数名是否正确（无拼写错误）
- 是否在 app.py 中导入和注册 v1 Blueprint
- 代码是否符合 PEP8 规范
- 错误处理是否完善

**二、功能验证（8 项）**:
- 本地测试通过（curl 测试）
- 响应格式正确
- 响应内容符合预期
- 错误处理正常
- 域名测试通过
- Nginx 转发正确
- SSL 证书有效
- 性能达标（响应时间<2s）

**三、文档同步（12 项）**:
- 更新接口文档 V2.0
- 包含请求参数说明、请求示例、响应示例、错误码说明
- 标注小程序调用路径和后端实际路径
- 更新接口注册表
- 填写接口设计文档
- 开发后验收已填写、开发者已签名

**四、提交确认（6 项）**:
- Git 提交信息规范
- 提交文件完整、已推送
- 接口设计文档、测试报告准备就绪
- 已预约评审会时间

**自检要求**:
- 36 项全部通过才能提交评审
- 自检清单必须开发者签名
- 自检清单作为评审材料归档

**模板**: `docs/接口开发自检清单.md`

### 2.4 接口注册表制度

- 所有接口必须注册到 `docs/接口注册表.yaml`
- 统一路径：`/api/v1/*`（小程序调用）
- 先注册后开发

### 2.5 接口路径规范

- 小程序调用：`/api/v1/模块名/功能`
- 后端 Phase: `/api/vX/...` (X=2,3,4,6,7,9...)
- 必须添加 v1 兼容 Blueprint

## 第三章 评审与测试制度

### 3.1 评审会议事规则

**评审会类型**:
- 开发前评审：按需，30-60 分钟
- 开发后验收：按需，30-60 分钟
- 每日站会：每日 9:00，15 分钟
- 周评审会：每周五 15:00，60 分钟

**评审流程**:
- 开发前评审：开发者介绍→评审人提问→评审讨论→评审结论→签字确认
- 开发后验收：开发者演示→评审人实测→评审讨论→验收结论→签字确认

**评审结论**: 通过/有条件通过/不通过

**争议解决**: 技术分歧→铸基决策，前端分歧→匠心决策，测试分歧→铁壁决策，重大分歧→总裁决策

**模板**: `docs/评审会议事规则_V1.0.md`

### 3.2 测试执行流程

**测试分级**:
- L1 单元测试（灵指）：开发完成后，通过率 100%
- L2 集成测试（铁壁）：模块完成后，通过率≥95%
- L3 E2E 测试（铁壁）：部署前必须，P0 100% 通过、总≥95%
- L4 性能测试（铁壁）：重大版本前，响应<2s

**必测流程**: 登录、会员购买、文书生成、AI 对话

**模板**: `docs/测试执行流程_V1.0.md`

### 3.3 Bug 跟踪流程

**Bug 分级**:
- P0 致命：系统崩溃，5 分钟响应，2 小时解决
- P1 严重：重要功能异常，30 分钟响应，24 小时解决
- P2 一般：一般功能异常，2 小时响应，72 小时解决
- P3 轻微：UI 问题，24 小时响应，1 周解决

**处理流程**: 发现→记录→分配→修复→验证→关闭

**模板**: `docs/Bug 跟踪流程_V1.0.md`

### 3.4 E2E 测试用例

**测试用例**: 登录流程、会员购买流程、文书生成流程、AI 对话流程

**模板**: `docs/E2E 测试用例_V1.0.md`

## 第七章 红线与违规处理

### 7.1 红线（违反=严重失误）

1. **未经评审先开发** - 书面警告 + 重新评审
2. **未经测试先上线** - 通报批评 + 绩效扣分
3. **擅自修改生产** - 严重警告 + 权限回收
4. **文档不同步** - 口头警告 + 立即整改
5. **自检清单未填写** - 口头警告 + 立即补填
6. **现场演示测试失败** - 重新开发 + 重新评审
7. **Bug 隐瞒不报** - 通报批评 + 绩效扣分
8. **培训考核不通过** - 补考 + 口头警告
9. **泄露敏感信息** - 严重警告 + 辞退
10. **直接 main 分支开发** - 书面警告

### 7.2 违规处理流程

发现违规→记录违规→调查核实→确定处理→执行处理→记录归档→申诉处理
