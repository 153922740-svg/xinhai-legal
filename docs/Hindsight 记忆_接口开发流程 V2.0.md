## 核心流程（必须严格执行）

### 一、两阶段评审制度

**阶段 1: 开发前评审**
```
填写接口设计文档 → COO 初审 → 技术评审 (铸基) → 前端评审 (匠心) → 测试评审 (铁壁) → COO 审批 → 开始开发
```

**阶段 2: 开发后验收**
```
开发完成 → 开发者 36 项自检 → 提交代码 → 评审会现场演示 → 实际测试 → 评审通过 → 代码合并
```

**关键改进**:
- 从"先开发后补票" → "先设计后开发"
- 从"形式评审" → "实测评审"（现场演示 + 实际测试）
- 从"凭感觉检查" → "清单式检查"（36 项自检）

---

### 二、接口设计文档制度

**原则**: 先设计，后开发。未经设计评审，不得开始开发。

**文档内容**:
1. **基本信息**: 接口名称、路径 (`/api/v1/模块名/功能`)、方法、Phase、负责人
2. **请求参数**: 参数名、类型、必填、默认值、说明、示例
3. **响应格式**: 成功响应、错误响应、错误码表
4. **后端设计**: v1 兼容 Blueprint 代码、处理函数调用关系
5. **测试设计**: 测试用例表、测试命令
6. **评审记录**: 开发前评审意见、开发后验收结果

**模板**: `docs/接口设计文档模板.md`

**违规处理**: 未经设计评审直接开发 = 书面警告 + 重新评审

---

### 三、36 项自检清单制度

**原则**: 开发完成后，开发者必须先自检，全部通过才能提交评审。

#### 一、代码实现检查（10 项）
- [ ] 是否添加了 `phaseX_v1_bp` Blueprint
- [ ] Blueprint 前缀是否为 `/api/v1/模块名`
- [ ] 路由路径是否正确（与接口设计文档一致）
- [ ] 请求方法是否正确（GET/POST/PUT/DELETE）
- [ ] v1 路由函数是否调用原处理函数
- [ ] 函数名是否正确（无拼写错误）
- [ ] 是否在 app.py 中导入 v1 Blueprint
- [ ] 是否在 app.py 中注册 v1 Blueprint
- [ ] 代码是否符合 PEP8 规范
- [ ] 错误处理是否完善

#### 二、功能验证检查（8 项）
- [ ] 本地测试通过（curl 测试）
- [ ] 响应格式正确（code/data/message）
- [ ] 响应内容符合预期
- [ ] 错误处理正常
- [ ] 域名测试通过（xinclaw.xhacca.cn）
- [ ] Nginx 转发正确
- [ ] SSL 证书有效
- [ ] 性能达标（响应时间<2s）

#### 三、文档同步检查（12 项）
- [ ] 更新了 `docs/接口文档_V2.0.md`
- [ ] 包含请求参数说明
- [ ] 包含请求示例
- [ ] 包含响应示例
- [ ] 包含错误码说明
- [ ] 标注小程序调用路径
- [ ] 标注后端实际路径
- [ ] 更新了 `docs/接口注册表.yaml`
- [ ] 接口 ID 唯一
- [ ] 填写了 `docs/接口设计文档_XXX.md`
- [ ] 开发后验收已填写
- [ ] 开发者已签名

#### 四、提交确认（6 项）
- [ ] Git 提交信息规范（`<type>(<scope>): <subject>`）
- [ ] 提交的文件完整
- [ ] 已推送到远程仓库
- [ ] 接口设计文档准备就绪
- [ ] 测试报告准备就绪
- [ ] 已预约评审会时间

**自检要求**:
- 36 项全部通过才能提交评审
- 自检清单必须开发者签名
- 自检清单作为评审材料归档

**模板**: `docs/接口开发自检清单.md`

---

### 四、评审方式

**开发前评审**: 文档评审（会议或在线）
- 评审接口设计文档
- 确认路径、参数、响应格式
- 评审通过才能开始开发

**开发后验收**: 现场演示 + 实际测试（必须）
- 开发者现场运行接口
- 评审人观察测试结果
- 使用 curl 或 Postman 实际测试
- 测试通过才能验收

**现场演示要求**:
```bash
# 本地测试
curl -X POST "http://localhost:8081/api/v1/模块名/功能" \
  -H "Content-Type: application/json" \
  -d '{"param":"value"}'

# 域名测试
curl -X POST "https://xinclaw.xhacca.cn/api/v1/模块名/功能" \
  -H "Content-Type: application/json" \
  -d '{"param":"value"}'
```

---

### 五、违规处理

| 违规类型 | 处理措施 |
|----------|----------|
| 未经评审先开发 | 书面警告 + 重新评审 |
| 评审未通过擅自上线 | 通报批评 + 绩效扣分 |
| 自检清单未填写 | 口头警告 + 立即补填 |
| 现场演示测试失败 | 重新开发 + 重新评审 |

**红线**: 未经评审的接口一律不准上线！

5. **合并前**: 必须更新所有文档，保持代码文档一致
