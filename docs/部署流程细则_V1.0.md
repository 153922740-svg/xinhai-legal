## 二、部署流程

### 2.2 部署流程

**开发环境部署**:
```
开发完成 → 本地测试 → 部署到开发环境 → 验证
```

**测试环境部署**:
```
PR 合并 → 自动部署 → 运行测试 → 测试报告
```

**预发布环境部署**:
```
测试通过 → COO 审批 → 部署到预发布 → 验证测试 → 总裁验收
```

**生产环境部署**:
```
预发布验证 → 总裁审批 → 部署到生产 → 线上验证 → 监控
```

## 五、回滚流程

### 5.1 回滚场景

- 上线后发现 P0 Bug
- 功能严重不符合需求
- 性能问题严重影响用户
- 安全漏洞

### 5.2 回滚决策

**决策人**: COO + 总裁

**决策依据**:
- Bug 严重程度
- 影响范围
- 修复难度
- 修复时间

### 5.3 回滚流程

```
发现需要回滚 → COO 审批 → 停止服务 → 恢复备份 → 验证 → 重启服务 → 通知用户
```

### 5.4 回滚命令

```bash
# 1. 停止服务
pkill -f "python.*app.py"

# 2. 恢复备份
rm -rf /home/admin/xinhai_legal_api
cp -r /home/admin/xinclaw-backup/api-20260518 /home/admin/xinhai_legal_api

# 3. 重启服务
cd /home/admin/xinhai_legal_api
source venv/bin/activate
nohup python app.py &

# 4. 验证
curl http://localhost:8081/api/v1/health
```

## 七、部署纪律

### 7.1 部署要求

- **计划性**: 提前制定部署计划
- **规范性**: 严格按照流程部署
- **可追溯**: 部署记录完整
- **可回滚**: 备份和回滚方案完备

### 7.2 禁止事项

- ❌ 未经审批部署生产环境
- ❌ 直接在生产环境修改代码
- ❌ 无备份部署
- ❌ 无验证部署
- ❌ 无回滚方案部署

### 7.3 违规处理

| 违规行为 | 处理措施 |
|----------|----------|
| 未经审批部署生产 | 严重警告 + 绩效扣分 |
| 直接修改生产代码 | 严重警告 |
| 无备份部署 | 书面警告 |
| 部署不验证 | 口头警告 |
