**版本**: V2.0  
**生效日期**: 2026-05-18  
**负责人**: COO  
**适用范围**: 所有技术开发工作

## 第一章 总则

### 1.1 制度目的

1. **规范开发流程** - 确保开发工作有序、高效
2. **保证代码质量** - 建立质量标准和审查机制
3. **避免重复问题** - 从接口路径问题等历史问题中吸取教训
4. **提升协作效率** - 明确职责分工和协作流程

### 1.2 核心原则

1. **PRD 驱动** - 所有开发必须基于 PRD 文档
2. **评审先行** - 关键节点必须经过评审
3. **测试保障** - 所有功能必须经过测试
4. **文档同步** - 代码与文档必须同步更新

### 1.3 组织架构

| 角色 | 职责 | 人员 |
|------|------|------|
| **总裁** | 战略决策、最终审批 | 用户 |
| **COO** | 项目管理、流程执行、质量把控 | 心海 |
| **架构师（铸基）** | 技术架构、接口评审、代码审查 | Agent |
| **产品官（蓝图）** | 需求分析、PRD 编写、原型设计 | Agent |
| **设计官（匠心）** | UI 设计、前端开发、接口调用评审 | Agent |
| **编码官（灵指）** | 功能开发、单元测试 | Agent |
| **测试官（铁壁）** | 测试用例、E2E 测试、质量报告 | Agent |

## 第二章 接口管理制度

### 2.1 接口设计文档制度（新增 - 最高优先级）

**原则**: 先设计，后开发。未经设计评审，不得开始开发。

**流程**:
```
填写接口设计文档 → 评审会通过 → 开始开发
```

**接口设计文档内容**:
1. **基本信息**: 接口名称、路径、方法、Phase、负责人
2. **请求参数**: 参数名、类型、必填、默认值、说明、示例
3. **响应格式**: 成功响应、错误响应、错误码
4. **后端设计**: v1 兼容路由代码、处理函数
5. **测试设计**: 测试用例、测试命令
6. **评审记录**: 开发前评审、开发后验收

**模板文件**: `docs/接口设计文档模板.md`

**违规处理**: 未经设计评审直接开发 = 书面警告 + 重新评审

---

### 2.2 接口评审制度

#### 评审组织

| 角色 | 人员 | 职责 |
|------|------|------|
| **组长** | COO | 组织评审、最终审批 |
| **技术评审** | 铸基（架构师） | 技术架构、路由设计 |
| **前端评审** | 匠心（设计官） | 接口调用、参数格式 |
| **测试评审** | 铁壁（测试官） | 测试用例、验收标准 |

#### 评审流程（两个阶段）

**阶段 1: 开发前评审**
```
开发提交接口设计文档 → COO 初审 → 技术评审 → 前端评审 → 测试评审 → COO 审批 → 开始开发
```

**阶段 2: 开发后验收**
```
开发完成 → 开发者自检 → 提交代码 → 现场演示 → 实际测试 → 评审通过 → 代码合并
```

#### 评审内容

1. **接口路径** - 是否为 `/api/v1/模块名/功能` 格式
2. **接口参数** - 类型、必填、默认值、验证规则
3. **接口响应** - 格式统一、错误码、分页信息
4. **安全性** - 认证、鉴权、SQL 注入、XSS 防护
5. **v1 兼容** - 是否添加 v1 兼容 Blueprint
6. **文档** - 接口文档、请求示例、响应示例
7. **测试** - 测试用例是否覆盖完整

#### 评审时限

| 接口类型 | 评审时限 | 审批人 |
|----------|----------|--------|
| 核心接口（登录/支付） | 24 小时 | COO |
| 重要接口（AI/文书） | 48 小时 | COO |
| 一般接口 | 72 小时 | COO 助理 |

#### 评审方式

**开发前评审**: 文档评审（会议或在线）
**开发后验收**: 现场演示 + 实际测试

**现场演示要求**:
- 开发者现场运行接口
- 评审人观察测试结果
- 测试通过才能验收

#### 违规处理

| 违规程度 | 处理措施 |
|----------|----------|
| 未经评审先开发 | 书面警告 + 重新评审 |
| 评审未通过擅自上线 | 通报批评 + 绩效扣分 |
| 自检清单未填写 | 口头警告 + 立即补填 |

**红线**: 未经评审的接口一律不准上线！

- [ ] 函数名是否正确（无拼写错误）
- [ ] 是否在 app.py 中导入 v1 Blueprint
- [ ] 是否在 app.py 中注册 v1 Blueprint
- [ ] 代码是否符合 PEP8 规范
- [ ] 错误处理是否完善

**二、功能验证检查（8 项）**
- [ ] 本地测试通过（curl 测试）
- [ ] 响应格式正确

- [ ] 错误处理正常
- [ ] 域名测试通过

- [ ] SSL 证书有效
- [ ] 性能达标（响应时间<2s）

**三、文档同步检查（12 项）**
- [ ] 更新了接口文档 V2.0
- [ ] 包含请求参数说明
- [ ] 包含请求示例
- [ ] 包含响应示例
- [ ] 包含错误码说明
- [ ] 标注小程序调用路径
- [ ] 标注后端实际路径
- [ ] 更新了接口注册表
- [ ] 接口 ID 唯一
- [ ] 填写了接口设计文档
- [ ] 开发后验收已填写
- [ ] 开发者已签名

**四、提交确认（6 项）**
- [ ] Git 提交信息规范
- [ ] 提交文件完整
- [ ] 已推送到远程仓库
- [ ] 接口设计文档准备就绪
- [ ] 测试报告准备就绪
- [ ] 已预约评审会时间

**自检要求**:
- 36 项全部通过才能提交评审
- 自检清单必须开发者签名
- 自检清单作为评审材料归档

### 2.4 接口注册表制度

#### 注册要求

1. **所有接口必须注册** - 在接口注册表中登记
2. **统一路径规范** - 小程序调用路径为 `/api/v1/*`
3. **先注册后开发** - 未注册的接口不得开发

#### 注册表结构

```yaml
version: "1.0"
updated: "2026-05-18"
base_url: "https://xinclaw.xhacca.cn/api/v1"
interfaces:
  - id: auth_send_sms
    name: 发送验证码
    path: /api/v1/auth/send_sms
    method: POST
    phase: Phase 4
    handler: send_sms_code
    status: active
    v1_compat: true
```

#### 维护责任

- **负责人**: COO
- **更新时机**: 新增接口、修改接口、废弃接口
- **审核**: COO 审批后更新

---

### 2.5 接口路径规范

#### 统一规范

1. **小程序调用路径**: `/api/v1/模块名/功能`
2. **后端 Phase 路径**: `/api/vX/...`（X=2,3,4,6,7,9...）
3. **v1 兼容路由**: 必须添加 v1 兼容 Blueprint

#### 示例

```python
# ✅ 正确：添加 v1 兼容路由
phase4_v1_bp = Blueprint('phase4_v1', __name__, url_prefix='/api/v1/auth')

@phase4_v1_bp.route('/send_sms', methods=['POST'])
def v1_send_sms():
    return send_sms_code()

# ❌ 错误：只有 v4 路由，没有 v1 兼容
phase4_bp = Blueprint('phase4_user', __name__, url_prefix='/api/v4')
```

## 第三章 测试管理制度

### 3.1 测试分级

| 级别 | 类型 | 负责人 | 执行时机 |
|------|------|--------|----------|
| **L1** | 单元测试 | 灵指 | 开发完成后 |
| **L2** | 集成测试 | 铁壁 | 模块完成后 |
| **L3** | E2E 测试 | 铁壁 | 部署前必须 |
| **L4** | 性能测试 | 铁壁 | 重大版本前 |

### 3.2 E2E 测试要求

#### 必测流程

1. **登录流程** - 验证码发送、登录、Token 验证
2. **会员购买** - 套餐列表、订单创建、支付
3. **文书生成** - 模板列表、文书生成、下载
4. **AI 对话** - 发送消息、会话列表、历史查询

#### 验收标准

- [ ] 所有 P0 测试用例通过
- [ ] 测试通过率 ≥ 95%
- [ ] 无严重 Bug
- [ ] 性能达标（响应时间 < 2s）

#### 测试报告

每次测试必须生成报告，包含：
- 测试结果（通过/失败）
- 失败用例详情
- 问题汇总
- 改进建议

---

### 3.3 自动化测试

#### CI/CD 集成

```yaml
# .github/workflows/e2e-test.yml
name: E2E Test
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run E2E Tests
        run: pytest tests/e2e_test.py -v
```

#### 执行频率

- **代码提交**: 自动触发
- **每日构建**: 凌晨 2:00 执行
- **部署前**: 必须手动执行一次

## 第五章 代码管理制度

### 5.1 Git 工作流

#### 分支管理

```
main        - 生产环境分支（受保护）
staging     - 预发布分支
develop     - 开发分支
feature/*   - 功能分支
hotfix/*    - 紧急修复分支
```

#### 提交规范

```
<type>(<scope>): <subject>

type: feat|fix|docs|style|refactor|test|chore
scope: auth|member|payment|document|chat|...
subject: 简短描述（不超过 50 字）
```

**示例**:
```
feat(auth): 添加微信登录功能

- 实现微信授权登录
- 添加 openid 缓存
- 完善错误处理
```

### 6.1 环境管理

| 环境 | 用途 | 部署方式 | 负责人 |
|------|------|----------|--------|
