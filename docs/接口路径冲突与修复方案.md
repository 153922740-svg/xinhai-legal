| **小程序调用** | `/auth/send_sms` → 完整：`/api/v1/auth/send_sms` | - | ✅ |
| **Nginx 代理** | `/api/v1/` → `http://127.0.0.1:8081/api/v1/` | - | ✅ |
| **后端 Phase 4** | - | `/api/v4/sms/send` | ❌ **不匹配** |
| **后端 Phase 4** | - | `/api/v4/user/login/sms` | ❌ **不匹配** |

## 🔧 修复方案

### 方案 A: 修改 Nginx 配置（推荐 ⭐）

**优点**:
- 不修改后端代码
- 不影响现有 Phase 结构
- 快速生效

**缺点**:
- Nginx 配置复杂
- 需要维护路径映射规则

**实施步骤**:

```nginx
# 在 /etc/nginx/conf.d/xinclaw.conf 中添加

# Phase 4 认证接口映射 (v1 → v4)
location /api/v1/auth/ {
    rewrite ^/api/v1/auth/(.*)$ /api/v4/$1 break;
    proxy_pass http://127.0.0.1:8081;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

# Phase 2 会员接口映射 (v1/member → v2/membership)
location /api/v1/member/ {
    rewrite ^/api/v1/member/(.*)$ /api/v2/membership/$1 break;
    proxy_pass http://127.0.0.1:8081;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}

# Phase 2 Token 接口映射 (v1/token → v2/token)
location /api/v1/token/ {
    rewrite ^/api/v1/token/(.*)$ /api/v2/token/$1 break;
    proxy_pass http://127.0.0.1:8081;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}

# Phase 2 支付接口映射 (v1/payment → v2/payment)
location /api/v1/payment/ {
    rewrite ^/api/v1/payment/(.*)$ /api/v2/payment/$1 break;
    proxy_pass http://127.0.0.1:8081;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}

# Phase 3 AI 对话接口映射 (v1/chat → v3/chat)
location /api/v1/chat/ {
    rewrite ^/api/v1/chat/(.*)$ /api/v3/chat/$1 break;
    proxy_pass http://127.0.0.1:8081;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}

# Phase 3 文书接口映射 (v1/document → v3/document)
location /api/v1/document/ {
    rewrite ^/api/v1/document/(.*)$ /api/v3/document/$1 break;
    proxy_pass http://127.0.0.1:8081;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;

# Phase 3 合同审阅接口映射 (v1/contract → v3/contract)
location /api/v1/contract/ {

    proxy_set_header X-Real-IP $remote_addr;
}
```

### 方案 B: 修改后端 Blueprint 前缀

**优点**:
- 路径统一，易于理解
- 减少 Nginx 配置复杂度

**缺点**:
- 需要修改多个 Phase 文件
- 可能影响其他调用方
- 需要重启后端服务

**实施步骤**:

```python
# phase4_user_auth_api.py
# 修改前：
phase4_bp = Blueprint('phase4_user', __name__, url_prefix='/api/v4')

# 修改后：
phase4_bp = Blueprint('phase4_user', __name__, url_prefix='/api/v1')
```

需要修改的文件：
- phase2_member_api.py: `/api/v2` → `/api/v1`
- phase2_token_billing.py: `/api/v2/token` → `/api/v1/token`
- phase2_payment_wechat.py: `/api/v2/payment` → `/api/v1/payment`
- phase3_ai_chat_api.py: `/api/v3/chat` → `/api/v1/chat`
- phase3_document_generator.py: `/api/v3/document` → `/api/v1/document`

## ✅ 推荐方案：方案 A (Nginx 路径映射)

### 理由：
1. **最小改动**：只修改 Nginx 配置，不影响后端代码
2. **快速生效**：`nginx -s reload` 即可，无需重启后端
3. **向后兼容**：保留原有 Phase 结构，不影响其他调用方
4. **易于维护**：所有路径映射集中在 Nginx 配置中
